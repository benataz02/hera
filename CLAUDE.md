# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

HERA is a B2B SAP CPQ + integration platform for SME manufacturers on SAP Business One.
This repo is the **backbone milestone only**: sign in → act as a tenant → create a quote →
watch it sync to a real SAP B1 sandbox via an outbound-only on-prem agent, with idempotency
that makes a duplicate B1 document physically impossible. Configurator, historical analysis,
multi-entity autodiscovery, and n8n are explicitly out (see `INITIAL-SPEC.md`,
`.claude/plans/full-initial-spec-md-greenfield-project-squishy-shore.md`).

Posture is deliberately minimal. Shortcuts are marked with `// ponytail:` comments naming the
upgrade path; do not "fix" them without a reason that has actually fired.

## Commands

Package manager is **bun** (not npm). Uses **isolated install** — every workspace must list its
own direct deps; transitive resolution does not work here. Run from repo root unless noted.

```
docker compose up -d db          # just Postgres (dev: run server/web/agent with bun)
bun run db:migrate               # apply Drizzle migrations
bun run db:generate              # generate a new migration from schema changes
bun run db:studio                # Drizzle Studio
bun run auth:generate            # regenerate Better Auth Drizzle schema -> packages/db/src/schema/auth.ts

bun run dev:server               # Hono+oRPC+auth on :3000, serves SPA in prod
bun run dev:web                  # Vite SPA on :5173, proxies /rpc + /api to :3000
bun run dev:agent                # on-prem pull loop (needs B1_* env)

bun run seed:dev                 # seed dev@hera.test + acme org + agent token (server must be up)
bun run e2e                      # the one end-to-end self-check (see below)
bun run build:web                # build SPA into apps/web/dist
docker compose --profile full up # prod: Postgres + server + Caddy wildcard subdomains
```

No test framework. Verification is `scripts/e2e.ts` (assert-based). It needs Postgres + a running
server: `docker compose up -d db && bun run db:migrate && bun run dev:server &` then `bun run e2e`.
Part A exercises the real cloud loop; Part B runs the agent's delivery decision against a mock B1
(no sandbox needed). `scripts/live-b1.ts` runs the same path against a real B1 sandbox. Other
`scripts/*.ts` (`b1-ping`, `make-quote`, `notify-test`, `timing`) are throwaway manual probes.

## Architecture

Three apps + one shared db package, bun workspaces (`apps/*`, `packages/*`), no Turborepo.

- **`packages/db`** — Drizzle schema (`auth` generated by Better Auth, `tenant`, `outbox`, `quote`),
  a `pg` Pool wrapped by Drizzle (`client.ts`), and `listener.ts`.
- **`apps/server`** — one Bun process running Hono = Better Auth (`/api/auth/*`) + oRPC (`/rpc/*`) +
  static SPA. Exports `AppRouter` **type** consumed by web and agent for end-to-end typing.
- **`apps/web`** — Vite SPA, UI5 Web Components (`sap_horizon`), TanStack Router (file-based) +
  TanStack Query + oRPC client. Same-origin with the API so Better Auth cookies just work.
- **`apps/agent`** — stateless on-prem pull loop. The **only** thing that talks to SAP B1
  (`ServiceLayerClient` lives here). Dials cloud over outbound HTTPS — no inbound ports, no VPN.

### Two auth layers (`apps/server/src/orpc/base.ts`)
- `userProcedure` — Better Auth session for *who*; tenant = the **request subdomain** (`<slug>.<baseDomain>`,
  read from `x-forwarded-host`/`host`) resolved to an org and gated by membership (org = tenant). A forged
  host can only ever select an org you already belong to. `activeOrganizationId` no longer drives tenancy.
- `agentProcedure` — per-tenant **bearer token**, hashed (SHA-256) and matched against
  `tenant_integration.agentTokenHash`. Tenant comes from the credential, never an IP or session.

### The sync loop (the architecture proof — read before touching `quote`/`sync`/`agent`)
1. `quote.create` writes the `quote` row **and** the `outbox` row in **one transaction**, then
   `pg_notify` in that same tx. Both commit or neither — the intent to sync can never be lost.
2. `sync.pull` (agent only) long-polls: claims rows with `FOR UPDATE SKIP LOCKED` + a lease,
   incrementing `attempts` **at claim time**. Parks on `waitForNotify` (~25s), falls back to the
   timeout if a NOTIFY is missed — slower, never lost.
3. Agent (`apps/agent/src/sync.ts`, `processItem`) decides delivery by `attempts`:
   `attempts == 1` → POST directly (dup impossible). `attempts > 1` → **GET-before-POST**; if the
   record exists, just `ack`. A unique-key conflict resolves by GET+ack, never a blind re-POST.
   **Never `ack` without a confirmed result; never re-POST without a GET.**
4. `sync.ack` flips outbox+quote to `synced` and `pg_notify`s the browser. `sync.nack` →
   transient (lease backoff, redelivered) or permanent (dead-letter). All idempotent.
5. `quote.watch` is an oRPC SSE event-iterator; the browser subscribes and sees `syncing → synced`
   via the same LISTEN/NOTIFY doorbell.

### LISTEN/NOTIFY (`packages/db/src/listener.ts`)
**One** dedicated direct `pg.Client` runs `LISTEN`, fanned out in-process via a `Map<channel, Set>`.
Never one LISTEN connection per request. This requires a **direct** Postgres connection —
transaction-mode poolers (PgBouncer) break LISTEN. Keep `DATABASE_URL` direct.

## Divergences from the plan (don't "correct" these)
- Syncs **business partners** (`CardCode`), not sales orders. Idempotency anchor is a `CardCode`
  derived deterministically from the outbox `dedupKey` — not the planned `U_CpqExtId` UDF +
  HANA unique index (noted in `sync.ts` for real documents later).
- B1 credentials live **on the agent** (`.env`, next to Service Layer), never in the cloud DB.
  `tenant_integration` stores only the agent token hash + heartbeat — no encrypted-password column.

## Subdomain tenancy (`<slug>.<baseDomain>`)
- The subdomain **is** the tenant. `userProcedure` parses it (`apps/server/src/tenant.ts`) and joins
  `organization ⋈ member` to resolve the org id + role — the membership join is the boundary. Auth lives
  on the **apex** (login/signup/onboarding/select); tenant subdomains only serve the `_authed` app.
- `organization.slug` *is* the subdomain (no separate column, no random suffix). Onboarding lets the user
  pick it with a live `checkSlug` availability check. `crossSubDomainCookies` shares the session cookie
  across apex + subdomains.
- Web routing: apex `/` is the lobby dispatcher (0 orgs → onboarding, 1 → tenant subdomain, many → select);
  `_authed` on a tenant host reconciles Better Auth's active org via `setActive({ organizationSlug })`.
- Dev uses **`lvh.me`** — apex `http://lvh.me:5173`, tenant `http://acme.lvh.me:5173` (lvh.me + *.lvh.me
  resolve to 127.0.0.1; no hosts file, no certs). `localhost` can't be used: it behaves like a TLD, so a
  `.localhost` parent-domain cookie isn't shared with `*.localhost` and the session is lost on the subdomain.
  Base domain = `APP_BASE_DOMAIN` (server, `.env`) / `VITE_APP_BASE_DOMAIN` (web, `apps/web/.env.development`).
  The Vite proxy keeps the tenant Host (`changeOrigin:false`); e2e signals its tenant via `x-forwarded-host`.
